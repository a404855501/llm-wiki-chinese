# RAG 对比 LLM Wiki

## 是什么

RAG（检索增强生成，Retrieval-Augmented Generation）和 LLM Wiki 是两种让 LLM 利用文档知识的范式。它们有根本性的区别。

## 核心对比

| 维度 | RAG | LLM Wiki |
|------|-----|----------|
| **知识存在形式** | 原始文档片段 | 编译后的结构化 wiki |
| **查询时行为** | 检索 → 拼凑 → 生成 | 读取 → 综合 → 生成 |
| **知识积累** | 无——每次从头发现 | 有——[[增量知识编译]] |
| **交叉引用** | 每次查询时临时发现 | 预先建立并维护 |
| **矛盾处理** | 每次查询时重新检测 | 预先标记和追踪 |
| **综合深度** | 受限于检索窗口 | 已综合多来源的演进观点 |
| **维护成本** | 低（只需索引） | 中（需要 LLM 维护 wiki） |
| **查询质量** | 随文档量增加而下降 | 随 wiki 丰富度增加而提升 |

## RAG 的问题

1. **无积累**：问一个需要综合五份文档的问题，LLM 必须每次找到并拼凑
2. **窗口限制**：检索窗口有限，复杂问题可能遗漏关键片段
3. **无交叉引用**：文档间的关系不会被持久化
4. **矛盾不可见**：不同文档的矛盾不会被系统性标记

## LLM Wiki 的优势

1. **[[持久复合产物]]**：wiki 随使用变强
2. **预编译**：知识编译一次，查询直接利用编译结果
3. **可见结构**：图形视图、索引、日志让知识形态可感知
4. **可健康检查**：可以系统性检查健康状态

## LLM Wiki 的代价

1. **需要维护**：LLM 必须持续更新 wiki
2. **间接层**：多了一层 wiki，不是直接查询原始文档
3. **Schema 依赖**：需要配置文件规范 LLM 行为
4. **初始成本**：搭建和调优需要时间

## 何时选择

- **RAG 适合**：文档集固定、查询简单、不需要积累
- **LLM Wiki 适合**：知识持续增长、需要综合分析、追求长期积累
